## 구현 내용

Java 콘솔 기반 커머스 플랫폼을 객체 지향 설계 원칙을 적용하여 구현했습니다.

### 1. 도메인 설계
각 클래스의 역할과 책임을 분리하여 설계했습니다.
* **Product** : 상품명, 가격(long), 설명, 재고수량 관리
* **Category** : 카테고리명과 `List<Product>` 관리. CommerceSystem이 상품을 직접 관리하는 것이 아닌 Category가 소유하도록 책임 분리
* **Customer** : 고객명, 이메일, 등급(Grade) 관리
* **Cart** : 장바구니 항목(상품명, 수량, 가격) 관리. 담을 당시 가격을 스냅샷으로 저장

### 2. 서비스 설계
CommerceSystem이 모든 콘솔 입출력과 흐름 제어를 담당하도록 설계했습니다.
* 도메인 객체(Product, Category)는 데이터 보관만 담당
* 출력/입력/분기는 CommerceSystem이 전담
* 메서드 단위로 역할 분리 (printCategory, enterCategory, printProducts, printCart 등)
* **AdminService** : 관리자 모드의 전반적인 흐름 제어 및 인증 로직 담당

### 3. 장바구니 및 주문 기능
* 상품 선택 시 장바구니 추가 여부 확인
* 재고 확인 후 부족 시 경고 메시지 출력
* 동일 상품 담을 시 수량 증가 (재고 초과 방지)
* 장바구니에 상품이 있을 때만 주문 관리 메뉴 출력
* 주문 취소 시 장바구니 초기화

### 4. 관리자 인증 기능 (도전과제 Lv 2)
* 메인 메뉴에서 6번 입력 시 관리자 인증 모드로 진입합니다.
* 지정된 고정 비밀번호와 사용자의 입력값이 일치할 경우에만 관리자 메뉴(`runAdminMenu`)가 활성화됩니다.
* **보안 및 실패 제한 로직**: 비밀번호 오입력 시 실패 횟수가 누적되며, **연속 3회 실패 시** 인증이 강제 종료되고 메인 메뉴로 안전하게 복귀(`return`)하도록 제어 흐름을 설계했습니다.

### 5. 예외 처리 및 입력 유틸리티
* `InputHandle` 유틸 클래스로 메뉴 선택 등의 숫자 입력 예외를 한 곳에서 관리합니다.
* 숫자 외 입력 시 `InputMismatchException` 처리 및 버퍼 비우기 진행
* 범위 벗어난 번호 입력 시 올바른 범위 안내 후 재입력 요청

## 어려웠던 점
* CommerceSystem이 `List<Product>`를 직접 관리하던 구조에서 Category가 상품 목록을 소유하도록 책임을 분리하는 과정이 어려웠습니다. 단순히 클래스를 나누는 것이 아니라 누가 데이터를 소유하고, 누가 그 데이터를 사용하는지 명확히 구분하는 것이 중요하다는 것을 배웠습니다.

* 관리자 인증 기능을 추가하는 과정에서 `Scanner`를 여러 곳에서 사용하며 입력이 정상적으로 동작하지 않는 문제를 겪었습니다. 처음에는 `Scanner` 자체의 문제라고 생각했지만, 원인을 분석하는 과정에서 `System.in`과 `Scanner`의 관계, 그리고 입력 처리 방식에 대한 이해가 부족했다는 점을 확인할 수 있었습니다. 문제를 해결하며 콘솔 애플리케이션의 입력 흐름을 설계하는 것 역시 중요한 요소임을 배웠습니다.

## 배운 점
### 1. 객체지향 설계와 단일 책임 원칙(SRP)
클래스를 단순히 기능별로 나누는 것이 아니라, 각 객체가 하나의 책임에 집중하도록 설계하는 것이 유지보수성과 확장성에 큰 영향을 준다는 점을 배웠습니다. 특히 `Category`가 상품 데이터를 관리하도록 변경하면서 역할과 책임을 명확하게 분리하는 경험을 할 수 있었습니다.

### 2. 자원 관리와 입력 처리의 중요성
`try-with-resources`를 사용하며 자원의 생명주기를 안전하게 관리하는 방법을 익혔습니다. 또한 `Scanner`의 `nextInt()`, `next()`, `nextLine()`의 차이와 입력 버퍼 처리 방식을 학습하면서, 입력 처리 로직을 일관성 있게 설계하는 것이 프로그램 안정성에 중요하다는 점을 이해하게 되었습니다.
